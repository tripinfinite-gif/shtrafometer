# Личный кабинет Штрафометр — План реализации

> Дата: 2026-04-12
> Статус: ПЛАНИРОВАНИЕ

---

## 1. Концепция

Личный кабинет — центральный хаб для клиента. Вход по телефону (SMS OTP), минимум трения. Пользователь пришёл с проверки → регистрация за 15 секунд → выбор услуги → оплата → автоматическое выполнение.

### Ключевые принципы
- **Time-to-value < 3 минуты**: от проверки до оплаченной услуги
- **Progressive profiling**: сначала телефон + имя, email позже (когда нужно доставить документы)
- **Одна кнопка**: главное действие всегда очевидно
- **Zero empty states**: кабинет сразу показывает результаты проверки и рекомендации

## 2. Пользовательские сценарии (User Flows)

### Flow A — Автоматическая услуга (PDF-отчёт, автофикс)
```
Проверка сайта на главной
  → Кнопка "Получить PDF-отчёт" / "Исправить нарушения"
  → Модалка регистрации (имя + телефон, SMS код)
  → Личный кабинет (услуга уже в корзине)
  → Кнопка "Оплатить" → YooKassa
  → Webhook: payment.succeeded
  → Автоматическая генерация PDF / запуск автофикса
  → Уведомление в кабинете + push на email (если указан)
  → Документы доступны в кабинете для скачивания
```

### Flow B — Консультация (ручная услуга)
```
Проверка сайта на главной
  → Кнопка "Заказать консультацию"
  → Регистрация (имя + телефон)
  → Личный кабинет → заявка на консультацию создана (статус: "Ожидает подтверждения")
  → Админу падает уведомление
  → Админ назначает дату/время, выставляет счёт
  → Клиент видит в кабинете: дата + кнопка "Оплатить"
  → После оплаты: статус "Оплачено, ждите в назначенное время"
  → После консультации: статус "Выполнено", документы прикреплены
```

### Flow C — Возврат пользователя
```
Пользователь заходит на сайт повторно
  → Кнопка "Войти" (в хедере)
  → Ввод телефона → SMS код
  → Кабинет: все предыдущие проверки, заказы, документы
  → Рекомендации на основе последней проверки
```

## 3. Структура страниц личного кабинета

```
/cabinet/                     — Дашборд (главная кабинета)
/cabinet/sites/               — Мои сайты (список проверенных доменов)
/cabinet/sites/[domain]/      — Детали сайта (история проверок, нарушения, динамика)
/cabinet/orders/              — Мои заказы (все услуги)
/cabinet/orders/[id]/         — Детали заказа (статус, документы, оплата)
/cabinet/documents/           — Мои документы (PDF-отчёты, чеки, акты)
/cabinet/settings/            — Настройки (email, уведомления)

/auth/login/                  — Вход (телефон + SMS)
/auth/register/               — Регистрация (имя + телефон + SMS)
```

### 3.1 Дашборд (`/cabinet/`)

Первый экран после входа. Задача — показать ценность и подтолкнуть к действию.

**Блоки:**
1. **Приветствие**: "Здравствуйте, {имя}!" + количество проверенных сайтов
2. **Карточка последней проверки** (если есть):
   - Домен, дата, кол-во нарушений, сумма штрафов
   - Прогресс-бар: "X из Y нарушений исправлено"
   - CTA: "Исправить нарушения" или "Скачать отчёт"
3. **Активные заказы** (если есть):
   - Статус каждого заказа с таймлайном
   - Кнопка действия (оплатить / скачать / посмотреть)
4. **Быстрые действия**:
   - "Проверить новый сайт"
   - "Заказать PDF-отчёт"
   - "Записаться на консультацию"
5. **Баннер email** (если не указан):
   - "Укажите email, чтобы получать отчёты, чеки и уведомления о новых нарушениях"

### 3.2 Мои сайты (`/cabinet/sites/`)

Все проверенные домены пользователя. Каждый сайт — карточка:
- Домен + favicon
- Последняя проверка: дата + кол-во нарушений
- Светофор статуса: зелёный / жёлтый / красный
- Кнопка: "Проверить снова" / "Посмотреть детали"

### 3.3 Детали сайта (`/cabinet/sites/[domain]/`)

- **График динамики**: нарушения по времени (было 12 → стало 3)
- **Текущие нарушения**: таблица с иконками серьёзности
- **Рекомендации**: какие услуги закрывают какие нарушения
- **История проверок**: все проверки с датами
- **CTA внизу**: "Исправить всё за X руб" / "Получить подробный отчёт"

### 3.4 Мои заказы (`/cabinet/orders/`)

Таблица/список карточек:
- Дата, услуга, сайт, сумма, статус
- Цветовая кодировка статусов:
  - Новый (синий) → Оплачен (жёлтый) → В работе (оранжевый) → Выполнен (зелёный)
  - Отменён (серый)

### 3.5 Детали заказа (`/cabinet/orders/[id]/`)

- Вся информация о заказе
- **Таймлайн выполнения**: шаги с отметками времени
- **Документы**: скачать PDF, чек, акт
- **Для автофикса**: список исправлений + статус каждого
- **Для консультации**: дата/время + кнопка "Перенести"
- **Кнопка оплаты** (если не оплачен)

### 3.6 Мои документы (`/cabinet/documents/`)

Централизованное хранилище:
- PDF-отчёты
- Чеки оплаты (от YooKassa)
- Акты выполненных работ
- Фильтр по типу документа и сайту

### 3.7 Настройки (`/cabinet/settings/`)

- **Email**: добавление/изменение (с подтверждением)
- **Уведомления**: email/SMS о статусах заказов
- **Данные компании**: для актов и чеков (ИНН, название — опционально)

## 4. Модель данных (новые таблицы)

### 4.1 Таблица `users`

```sql
CREATE TABLE users (
  id TEXT PRIMARY KEY,                    -- nanoid
  created_at TIMESTAMPTZ DEFAULT NOW(),
  name TEXT NOT NULL,
  phone TEXT NOT NULL UNIQUE,             -- формат: +7XXXXXXXXXX
  email TEXT,                             -- добавляется позже (progressive profiling)
  email_verified BOOLEAN DEFAULT FALSE,
  company_name TEXT,                      -- для актов (опционально)
  company_inn TEXT,                       -- для актов (опционально)
  avatar_url TEXT,                        -- генерируем из инициалов (не загружаемый)
  last_login_at TIMESTAMPTZ,
  login_count INTEGER DEFAULT 0
);

CREATE UNIQUE INDEX idx_users_phone ON users (phone);
CREATE INDEX idx_users_email ON users (email) WHERE email IS NOT NULL;
```

### 4.2 Таблица `otp_codes` (SMS-коды)

```sql
CREATE TABLE otp_codes (
  id TEXT PRIMARY KEY,
  phone TEXT NOT NULL,
  code TEXT NOT NULL,                     -- 4-значный код
  purpose TEXT NOT NULL DEFAULT 'login',  -- 'login' | 'email_verify'
  created_at TIMESTAMPTZ DEFAULT NOW(),
  expires_at TIMESTAMPTZ NOT NULL,        -- created_at + 5 минут
  used BOOLEAN DEFAULT FALSE,
  attempts INTEGER DEFAULT 0              -- защита от брутфорса (макс 3)
);

CREATE INDEX idx_otp_phone ON otp_codes (phone, created_at DESC);
```

### 4.3 Таблица `user_sessions`

```sql
CREATE TABLE user_sessions (
  id TEXT PRIMARY KEY,                    -- session token (nanoid, 32 chars)
  user_id TEXT NOT NULL REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  expires_at TIMESTAMPTZ NOT NULL,        -- +30 дней
  ip TEXT,
  user_agent TEXT
);

CREATE INDEX idx_sessions_user ON user_sessions (user_id);
CREATE INDEX idx_sessions_expires ON user_sessions (expires_at);
```

**Почему не JWT для пользователей**: JWT нельзя отозвать. С серверными сессиями можем:
- Показать "активные устройства" в настройках
- Принудительно разлогинить (безопасность)
- Точно контролировать время жизни

### 4.4 Модификация таблицы `orders`

```sql
-- Добавляем связь с пользователем
ALTER TABLE orders ADD COLUMN user_id TEXT REFERENCES users(id);
ALTER TABLE orders ADD COLUMN payment_id TEXT;           -- ID платежа YooKassa
ALTER TABLE orders ADD COLUMN payment_status TEXT;       -- pending | succeeded | canceled
ALTER TABLE orders ADD COLUMN paid_at TIMESTAMPTZ;
ALTER TABLE orders ADD COLUMN completed_at TIMESTAMPTZ;
ALTER TABLE orders ADD COLUMN scheduled_at TIMESTAMPTZ;  -- для консультаций: дата/время

CREATE INDEX idx_orders_user ON orders (user_id);
CREATE INDEX idx_orders_payment ON orders (payment_id);
```

### 4.5 Таблица `documents`

```sql
CREATE TABLE documents (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id),
  order_id TEXT REFERENCES orders(id),
  type TEXT NOT NULL,                     -- 'report' | 'receipt' | 'act' | 'certificate'
  title TEXT NOT NULL,
  file_url TEXT NOT NULL,                 -- Vercel Blob / S3 URL
  file_size INTEGER,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_documents_user ON documents (user_id);
CREATE INDEX idx_documents_order ON documents (order_id);
```

### 4.6 Таблица `user_sites` (связь пользователь-домен)

```sql
CREATE TABLE user_sites (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id),
  domain TEXT NOT NULL,
  added_at TIMESTAMPTZ DEFAULT NOW(),
  last_check_at TIMESTAMPTZ,
  last_violations INTEGER DEFAULT 0,
  last_max_fine INTEGER DEFAULT 0,
  monitoring_enabled BOOLEAN DEFAULT FALSE,
  UNIQUE(user_id, domain)
);

CREATE INDEX idx_user_sites_user ON user_sites (user_id);
CREATE INDEX idx_user_sites_domain ON user_sites (domain);
```

## 5. Аутентификация (SMS OTP)

### 5.1 Провайдер SMS

**Рекомендация: SMS.ru**
- Цена: ~2-3 руб/SMS (одна из самых низких)
- Простой HTTP API
- Регистрация с ИП/ООО
- Имя отправителя: "Shtrafometr" (нужно зарегистрировать)

**Альтернатива: SMSC.ru** — если SMS.ru не подойдёт.

### 5.2 Flow регистрации

```
1. Пользователь вводит имя + телефон
2. POST /api/auth/send-code { phone, name }
   - Проверяем rate limit: макс 3 SMS на номер за 10 минут
   - Генерируем 4-значный OTP
   - Сохраняем в otp_codes (expires: +5 мин)
   - Отправляем SMS через SMS.ru API
   - Возвращаем { success: true, retryAfter: 60 }
3. Пользователь вводит код
4. POST /api/auth/verify-code { phone, code }
   - Проверяем OTP (не истёк, не использован, attempts < 3)
   - Если пользователь не существует → создаём (INSERT users)
   - Если существует → обновляем last_login_at
   - Создаём сессию в user_sessions
   - Устанавливаем cookie: user_session={sessionId} (httpOnly, secure, 30 дней)
   - Возвращаем { success: true, user: {...}, isNewUser: bool }
```

### 5.3 Flow входа

Тот же, но без поля "имя" (определяем по существующему телефону).

### 5.4 Защита от злоупотреблений

- **Rate limit по IP**: макс 5 запросов SMS в 10 мин
- **Rate limit по телефону**: макс 3 запроса в 10 мин
- **Блокировка после 5 неверных кодов**: бан телефона на 1 час
- **Минимальный интервал**: 60 сек между отправками на один номер
- **OTP одноразовый**: помечаем used=true сразу после успешной верификации

### 5.5 Сессии

- Cookie `user_session` — httpOnly, secure, SameSite=Lax, path=/
- Время жизни: 30 дней (обновляется при каждом визите)
- Proxy (бывший middleware) проверяет cookie для `/cabinet/*` и `/api/cabinet/*`
- Если сессия невалидна → редирект на `/auth/login`

## 6. Интеграция с YooKassa

### 6.1 Текущее состояние

Уже реализовано:
- `createPayment()` в [yookassa.ts](src/lib/yookassa.ts)
- Webhook handler в `/api/payment/webhook`
- Идемпотентность через SHA256 от orderId

### 6.2 Что нужно доработать

1. **Привязка платежа к user_id**: добавить user_id в metadata платежа
2. **Обновление order при успешной оплате**:
   - `payment_status = 'succeeded'`
   - `paid_at = NOW()`
   - Запуск автоматизации (генерация PDF / автофикс)
3. **Чеки**: YooKassa автоматически формирует чек (54-ФЗ) через receipt в createPayment — уже реализовано
4. **Return URL**: менять на `/cabinet/orders/{orderId}` вместо `/payment/success`
5. **Webhook расширение**:
   - При `payment.succeeded` → обновить order → запустить автоматизацию → уведомить клиента
   - При `payment.canceled` → обновить статус → уведомить клиента

### 6.3 Автоматизация после оплаты

| Продукт | Что происходит автоматически |
|---------|------------------------------|
| report | Генерация PDF → сохранение в documents → email клиенту |
| autofix-basic | Запуск fix-plan для топ-5 нарушений → отчёт в кабинет |
| autofix-std | Запуск fix-plan для всех нарушений → отчёт в кабинет |
| autofix-prem | autofix-std + создание заявки на ручную проверку |
| monitoring | Создание cron-задачи на ежемесячную проверку |
| consulting | Уведомление админу → ожидание назначения даты |

## 7. Фичи для повышения конверсии

### 7.1 Urgency & Scarcity (срочность)

- **Таймер на скидку**: "Скидка 20% на PDF-отчёт — осталось 23:47:12"
  - Запускается после первой проверки, 24 часа
  - Сохраняется в localStorage + на сервере для конкретного пользователя
- **"Проверку видели X человек за сегодня"**: социальное доказательство

### 7.2 Gamification (геймификация)

- **Прогресс-бар соответствия**: "Ваш сайт соответствует на 34%" → после исправлений → "87%"
  - Визуально мотивирует довести до 100%
  - Цвет: красный → жёлтый → зелёный
- **Бейджи/достижения**:
  - "Первая проверка" — после регистрации
  - "Безопасный сайт" — 0 критических нарушений
  - "Полное соответствие" — 0 нарушений
  - "Ответственный владелец" — подключён мониторинг

### 7.3 Social Proof (соц. доказательство)

- **Счётчик**: "Уже 1,247 сайтов проверено через Штрафометр"
- **Анонимизированные кейсы**: "Сайт интернет-магазина: было 15 нарушений, штрафы до 800К → после автофикса: 0 нарушений"
- **Отзывы**: виджет с отзывами клиентов в кабинете

### 7.4 Smart Recommendations (умные рекомендации)

- После проверки — персональный план действий:
  1. "Сначала исправьте критические нарушения (штраф до 500К) — PDF-отчёт от 1990 руб"
  2. "Затем автоисправление оставшихся — от 4990 руб"
  3. "Подключите мониторинг чтобы не пропустить новые нарушения — 490 руб/мес"
- **Upsell в момент покупки**: "Добавьте мониторинг за +490 руб/мес и мы предупредим о новых нарушениях"

### 7.5 Notification System (уведомления)

- **В кабинете**: колокольчик с badge (количество непрочитанных)
- **Email** (если указан):
  - Заказ оплачен → подтверждение
  - Отчёт готов → ссылка на скачивание
  - Автофикс завершён → сводка исправлений
  - Мониторинг: обнаружены новые нарушения
  - Рассылка: советы по комплаенсу (1 раз в 2 недели)
- **SMS** (только критичное):
  - Код входа
  - Заказ готов

### 7.6 Referral Program (реферальная программа) — v2

- "Пригласите коллегу — получите скидку 30% на следующий заказ"
- Уникальная реферальная ссылка в кабинете
- Трекинг: кто привёл, сколько заказов

### 7.7 Repeat Check Incentive (стимул повторной проверки)

- После исправлений: "Проверьте сайт снова бесплатно — убедитесь что всё в порядке"
- Показываем сравнение "до и после" — мощный визуал

### 7.8 Trust Signals (сигналы доверия)

- В кабинете и на странице оплаты:
  - "Оплата защищена" + логотипы (YooKassa, банковские карты, СБП)
  - "Данные вашего сайта в безопасности"
  - "Мы не храним пароли от серверов" (для автофикса)
  - Юридические реквизиты ООО «Инфологистик 24»

### 7.9 Onboarding Tour (обучающий тур)

- При первом входе: 3-шаговый интерактивный тур:
  1. "Здесь ваши проверенные сайты" (подсветка блока)
  2. "Здесь ваши заказы и документы" (подсветка)
  3. "Укажите email для получения отчётов" (фокус на поле)
- Можно пропустить ("Я разберусь сам")

### 7.10 Quick Actions (быстрые действия)

- Floating Action Button (FAB) на мобильном:
  - "Проверить сайт"
  - "Связаться с нами" (WhatsApp/Telegram)
- Sticky CTA на десктопе: если есть неоплаченные заказы

## 8. Progressive Profiling (сбор данных)

### Этап 1 — Регистрация (Day 0)
- **Собираем**: имя + телефон
- **Минимум трения**: 2 поля + SMS код

### Этап 2 — После первого заказа или на дашборде (Day 0-1)
- **Собираем**: email
- **Мотивация**: "Укажите email, чтобы получить отчёт и чек"
- **Баннер**: ненавязчивый, в верхней части дашборда
- **Не блокирует**: можно закрыть и указать позже

### Этап 3 — При заказе консультации или оформлении акта (Day 1+)
- **Собираем**: название компании, ИНН
- **Мотивация**: "Для оформления акта выполненных работ"
- **Опционально**: заполнять не обязательно для физлиц

### Этап 4 — Повторный визит (Day 7+)
- **Предлагаем**: подключить мониторинг
- **Мотивация**: "Законодательство меняется — мониторинг предупредит о новых нарушениях"

## 9. API Routes (новые)

### Auth
```
POST /api/auth/send-code      — Отправка SMS-кода
POST /api/auth/verify-code    — Проверка кода, создание сессии
POST /api/auth/logout         — Завершение сессии пользователя
```

### Cabinet
```
GET  /api/cabinet/me             — Текущий пользователь
PUT  /api/cabinet/me             — Обновление профиля (email, имя, компания)
GET  /api/cabinet/sites          — Мои сайты
GET  /api/cabinet/sites/[domain] — Детали сайта + история проверок
GET  /api/cabinet/orders         — Мои заказы
GET  /api/cabinet/orders/[id]    — Детали заказа
POST /api/cabinet/orders         — Создание заказа (выбор услуги)
POST /api/cabinet/orders/[id]/pay — Инициировать оплату
GET  /api/cabinet/documents      — Мои документы
GET  /api/cabinet/documents/[id] — Скачать документ
GET  /api/cabinet/notifications  — Уведомления
PUT  /api/cabinet/notifications/[id]/read — Пометить прочитанным
```

### Admin (расширение)
```
POST /api/admin/orders/[id]/schedule  — Назначить дату консультации
POST /api/admin/orders/[id]/complete  — Отметить заказ выполненным
POST /api/admin/orders/[id]/upload    — Загрузить документ к заказу
GET  /api/admin/users                 — Список пользователей (обновлённый)
```

## 11. Дизайн и UX

- Продолжаем Apple-стиль из текущего дизайна (тёмная тема, фиолетовый акцент #6C5CE7)

- **Десктоп**: левый sidebar (навигация) + основная область

- **Header**: имя пользователя, колокольчик уведомлений, аватар (инициалы)

- **StatusBadge**: цветные бейджи статусов заказов

- Все страницы mobile-first

- Swipe-жесты для навигации между разделами

## 12. Безопасность

- **OTP**: 4 цифры, TTL 5 мин, макс 3 попытки, rate limit
- **Сессии**: серверные, httpOnly cookies, 30 дней, ротация при подозрительной активности
- **Данные**: телефоны хранятся в нормализованном виде (+7XXXXXXXXXX)
- **Email verification**: ссылка-подтверждение при добавлении email
- **CORS**: API кабинета доступен только с shtrafometer.ru
- **Разделение**: admin auth (пароль + JWT) и user auth (SMS + сессии) — полностью независимые

## 16. Риски и митигация

| Риск | Митигация |
|------|-----------|
| SMS не доходят | Fallback: голосовой звонок с кодом (SMS.ru поддерживает) |
| Высокая стоимость SMS | Кеширование сессий 30 дней, минимум SMS |
| Юзеры не указывают email | Мотивация через ценность (отчёт, чек), повторные напоминания |
| Мало повторных визитов | Email-рассылка, мониторинг с уведомлениями |
| Сложность UX для ЦА (владельцы бизнеса 40+) | Крупный шрифт, минимум шагов, подсказки |
