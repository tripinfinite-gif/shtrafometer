## 1. Текущее состояние кодовой базы

### Аутентификация (src/lib/auth.ts)
- Один админ, пароль через bcryptjs (base64-encoded hash в env)
- JWT сессии через jose, cookie `admin_session`, 24ч TTL
- Proxy (src/proxy.ts) защищает `/admin/*` и `/api/admin/*`
- Proxy проверяет JWT, при невалидном — редирект на `/admin/login` или 401

### API-паттерны
- Все роуты — стандартные Next.js 16 App Router route handlers
- POST body → JSON parse → валидация → бизнес-логика → JSON response
- Rate limiting: in-memory per-IP (api/payment)
- Ошибки: разные HTTP коды (400, 401, 422, 429, 500)
- Fire-and-forget для email и логов (не блокируют ответ)

### Модель заказа
- Таблица `orders`: name, phone, email, site_url, domain, violations, price, status, product_type, check_result (JSONB)
- Нет таблицы users — каждый заказ standalone
- Статусы: new → in_progress → completed / cancelled
- Типы продуктов: fix, report, autofix-basic/std/prem, monitoring, consulting, email-lead

### Оплата (src/lib/yookassa.ts)
- YooKassa API: createPayment() с идемпотентностью (SHA256 от orderId)
- Webhook handler: payment.succeeded → обновление статуса заказа
- Receipt (54-ФЗ) формируется автоматически через YooKassa
- Return URL: /payment/success?order={orderId}

### UI
- Tailwind CSS 4, Apple-стиль, тёмная тема (#6C5CE7 акцент)
- Главная страница (app/page.tsx, 1186 строк): client component, формa + результаты
- Админка: AdminShell layout с nav + logout
- Email: Resend API, 3 шаблона (gate report, order confirmation, admin notification)

### Текущий flow покупки
1. Проверка сайта → результаты
2. Выбор продукта → форма (имя, телефон, email)
3. POST /api/payment → создание order + YooKassa payment → редирект на оплату
4. Webhook → обновление статуса

## 2. SMS-провайдеры

### SMS.ru (рекомендация)
- **API**: `https://sms.ru/sms/send` (GET/POST)
- **Авторизация**: api_id (один параметр)
- **Параметры**: api_id, to, msg, from_name, json=1
- **Ответ**: JSON с id и статусом
- **Цена**: от 4 коп/SMS
- **Имя отправителя**: до 11 символов, регистрация через ЛК (1-5 дней)
- **Node.js библиотека**: `node-sms-ru` (npm)

### SMSC.ru (альтернатива)
- Больше фич (flash SMS, MMS, входящие), но сложнее API
- Авторизация: login/password или apikey
- Для OTP избыточен

### Решение: прямой HTTP-вызов SMS.ru API
Не нужна npm-зависимость. Достаточно fetch():
```
GET https://sms.ru/sms/send?api_id=KEY&to=PHONE&msg=TEXT&json=1
```

## 3. Серверные сессии

### Выбор: PostgreSQL + cookie (vs JWT)
**Преимущества серверных сессий:**
- Можно отозвать (выход со всех устройств)
- Точный контроль TTL
- Нет утечки данных через token payload

### Токен сессии
- `crypto.randomUUID()` — встроенный в Node.js, быстрый (28M ops/sec), криптостойкий
- Не нужен nanoid (минус зависимость)
- 36 символов — достаточно для DB primary key

### Паттерн rolling sessions
- TTL: 30 дней
- Обновление expires_at при каждом визите (throttle: не чаще 1ч)
- Cookie: httpOnly, secure, sameSite=lax, path=/, maxAge=30d

### Защита роутов
- proxy.ts: проверка cookie → SELECT session → redirect if invalid
- Два набора матчеров: `/admin/*` (JWT) и `/cabinet/*` (server sessions)
- API: `/api/cabinet/*` — проверка сессии в каждом route handler через helper

## 4. OTP Best Practices

### Параметры
- 4 цифры (достаточно для SMS OTP, привычно пользователю)
- TTL: 5 минут
- Одноразовый: used=true после верификации
- Макс 3 попытки ввода на один код

### Rate limiting
- По телефону: макс 3 SMS за 10 мин
- По IP: макс 5 SMS за 10 мин
- Минимальный интервал: 60 сек между отправками на один номер
- После 5 неверных кодов: блокировка телефона на 1 час

### Хранение
- Plaintext в БД (4 цифры, TTL 5 мин — хеширование избыточно)
- Таблица otp_codes с автоматической очисткой expired записей

### Законодательство РФ
- Transactional SMS (OTP) — не требует маркетингового согласия
- Имя отправителя: регистрация у SMS-провайдера
- Текст: "Код входа: XXXX. Не сообщайте никому. Штрафометр"
- Данные хранить на территории РФ (152-ФЗ) — PostgreSQL в RU-регионе

## 5. Множественные заказы и проверки (дополнение)

### Проблема текущей модели
- Один заказ = один продукт = один сайт
- Нет корзины / мульти-заказа
- Нельзя заказать PDF + автофикс одновременно
- Нельзя проверить несколько сайтов из кабинета

### Решение: корзина заказов + мульти-сайт

**Вариант A: Отдельные заказы (рекомендация)**
- Каждый продукт — отдельный order с отдельной оплатой
- Проще реализовать, проще отслеживать статусы
- Пользователь может заказать несколько услуг последовательно
- В кабинете: список всех заказов с фильтрами

**Вариант B: Корзина**
- Сложнее: нужна таблица cart_items, логика group payment
- YooKassa поддерживает несколько items в одном payment (receipt.items[])
- Но статусы выполнения разные для каждой услуги → усложняет tracking

**Решение**: Вариант A + возможность "быстрого заказа" нескольких услуг.
На странице сайта в кабинете: кнопки для каждой услуги → каждая создаёт отдельный order → отдельная оплата.
В будущем (Phase 3+): bundle-скидки при покупке нескольких услуг.

### Мульти-сайт
- Таблица user_sites: пользователь → несколько доменов
- Каждый домен можно проверить из кабинета (POST /api/check)
- Результаты привязываются к user_sites
- Кнопка "Добавить сайт" в кабинете → ввод URL → проверка → результаты

### Повторные проверки
- Бесплатная повторная проверка после покупки услуги (для сравнения "до/после")
- Платная проверка нового сайта: бесплатно (проверка всегда бесплатна, платятся услуги)
- История всех проверок для каждого домена в user_sites

## 6. Рекомендация: ЛУЧШИЙ подход

### Архитектура
1. **Серверные сессии в PostgreSQL** (не JWT) — для отзываемости и контроля
2. **SMS.ru через прямой fetch()** — без лишних npm-зависимостей
3. **crypto.randomUUID()** для токенов — встроенный, быстрый
4. **Отдельная система auth** для пользователей (user_session cookie) — параллельно с admin auth (admin_session cookie)
5. **Отдельные заказы** (не корзина) — проще, надёжнее, каждый order с отдельной оплатой
6. **Мульти-сайт через user_sites** — пользователь привязывает несколько доменов

### Почему это лучший подход
- **Минимум зависимостей**: только pg (уже есть) + SMS.ru API (прямой fetch)
- **Совместимость**: не ломает текущую admin auth, работает параллельно
- **Масштабируемость**: серверные сессии → легко добавить "активные устройства", 2FA
- **Простота**: один order = один payment = один tracking — понятно и клиенту, и админу
- **UX**: проверка бесплатна → услуги платные → отдельные заказы с чёткими статусами
