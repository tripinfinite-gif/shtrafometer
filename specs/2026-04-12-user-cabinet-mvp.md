# Spec: Личный кабинет Штрафометр — MVP (Phase 1)

**Date:** 2026-04-12
**Author:** Владелец проекта
**Research:** thoughts/research/2026-04-12-user-cabinet.md

## Problem

Сейчас покупка услуг — одноразовый flow без аккаунта. Пользователь заполняет форму, оплачивает, и теряется. Нет личного кабинета, нет истории, нет привязки к пользователю. Это убивает retention, upsell и автоматизацию доставки услуг. Невозможно доставить PDF-отчёт в кабинет, показать историю проверок, предложить дополнительные услуги.

## Goal

Создать личный кабинет с SMS-аутентификацией, где пользователь:
1. Регистрируется за 15 секунд (имя + телефон + SMS код)
2. Видит свои сайты, заказы и документы
3. Заказывает и оплачивает услуги (несколько разных для разных сайтов)
4. Получает результат автоматически после оплаты

**Измеримый результат:** время от проверки до оплаты < 3 минут, конверсия регистрация→оплата > 15%.

## Scope

### In Scope (Phase 1 — MVP)

**Аутентификация:**
- Регистрация по имя + телефон + SMS OTP (4 цифры, 5 мин TTL)
- Вход по телефон + SMS OTP
- Серверные сессии в PostgreSQL (30 дней, rolling)
- Защита /cabinet/* и /api/cabinet/* через proxy.ts
- Rate limiting SMS: 3/10мин на номер, 5/10мин на IP

**Личный кабинет — страницы:**
- Дашборд: приветствие, последняя проверка, активные заказы, быстрые действия
- Мои сайты: список доменов пользователя с последними результатами
- Детали сайта: нарушения, рекомендации, кнопки заказа услуг
- Мои заказы: все заказы со статусами
- Детали заказа: таймлайн, документы, кнопка оплаты
- Настройки: добавление email

**Заказ услуг:**
- Заказ любой услуги для любого сайта из кабинета
- Несколько заказов для одного сайта (PDF + автофикс + мониторинг)
- Несколько сайтов у одного пользователя
- Каждый заказ — отдельная оплата через YooKassa
- Статусы: new → paid → in_progress → completed / cancelled

**Проверки из кабинета:**
- Проверка нового сайта (добавление в user_sites)
- Повторная проверка существующего сайта
- Проверка бесплатна, платятся только услуги
- История всех проверок для каждого домена

**Интеграция с главной:**
- Кнопка "Получить отчёт" / "Исправить" → регистрация → кабинет с предвыбранной услугой
- Кнопка "Войти" / "Кабинет" в навигации

**Оплата:**
- Инициирование оплаты из кабинета (POST /api/cabinet/orders/[id]/pay)
- Return URL → /cabinet/orders/[id] (вместо /payment/success)
- Webhook обновляет payment_status и paid_at

### Out of Scope (NOT doing in Phase 1)

- Автоматическая генерация PDF после оплаты (Phase 2)
- Страница "Мои документы" (Phase 2)
- Email-уведомления о статусах (Phase 2)
- Прогресс-бар соответствия, геймификация (Phase 4)
- Таймер скидки, social proof (Phase 4)
- Flow консультации с назначением даты (Phase 5)
- Мониторинг-подписка (Phase 5)
- Реферальная программа (Phase 5)
- Корзина / групповая оплата нескольких услуг одним платежом

## Acceptance Criteria

### Аутентификация
- [ ] AC1: Пользователь регистрируется вводя имя + телефон → получает SMS код → вводит код → попадает в кабинет
- [ ] AC2: Пользователь входит вводя телефон → SMS код → кабинет (имя уже известно)
- [ ] AC3: Неверный код 3 раза → "Код недействителен, запросите новый"
- [ ] AC4: Повторная отправка SMS не чаще 60 сек (кнопка с таймером)
- [ ] AC5: Неавторизованный доступ к /cabinet/* → редирект на /auth/login
- [ ] AC6: Сессия живёт 30 дней с rolling обновлением

### Кабинет
- [ ] AC7: Дашборд показывает имя, последнюю проверку, активные заказы, быстрые действия
- [ ] AC8: Список сайтов: все домены пользователя со статусом (нарушения/OK)
- [ ] AC9: Детали сайта: текущие нарушения + кнопки заказа каждой услуги
- [ ] AC10: Проверка нового сайта из кабинета → результаты → домен добавлен в user_sites
- [ ] AC11: Повторная проверка существующего сайта → обновление last_check результатов
- [ ] AC12: Список заказов: фильтр по статусу, для каждого — дата, услуга, сайт, сумма, статус
- [ ] AC13: Детали заказа: полная информация + таймлайн + кнопка оплаты (если не оплачен)
- [ ] AC14: Настройки: можно указать/изменить email

### Заказ и оплата
- [ ] AC15: Из деталей сайта можно заказать любую услугу (report, autofix-basic/std/prem, consulting)
- [ ] AC16: Можно заказать несколько разных услуг для одного сайта (каждая — отдельный order)
- [ ] AC17: Оплата из кабинета → редирект на YooKassa → возврат в детали заказа
- [ ] AC18: После успешной оплаты (webhook) → статус заказа меняется на paid → in_progress
- [ ] AC19: Для нескольких сайтов: услуги заказываются независимо для каждого

### Интеграция с главной
- [ ] AC20: На главной в хедере кнопка "Войти" (если не авторизован) / "Кабинет" (если авторизован)
- [ ] AC21: Кнопка "Получить отчёт" на странице результатов → если не авторизован: регистрация → кабинет с предвыбранной услугой

### Админка
- [ ] AC22: В списке заказов видно user_id и данные пользователя
- [ ] AC23: Можно менять статус заказа (как раньше)

## Constraints

### Технические
- Next.js 16 App Router, TypeScript, Tailwind CSS 4
- PostgreSQL через raw SQL (pg driver) — без ORM
- Серверные сессии (не JWT) для пользователей
- Admin auth (JWT) и User auth (server sessions) — полностью независимые системы
- SMS через SMS.ru API (прямой fetch, без npm-зависимостей)
- proxy.ts для защиты роутов

### Обратная совместимость
- Текущий flow покупки с главной страницы ДОЛЖЕН продолжать работать
- Админка НЕ затрагивается (кроме отображения user_id в заказах)
- Таблица orders: новые колонки (user_id, payment_id и т.д.) — nullable для старых записей
- Webhook handler: совместим с заказами и с user_id, и без

### Безопасность
- OTP: rate limit, одноразовость, TTL
- Сессии: httpOnly, secure, sameSite=lax
- Телефоны: нормализованный формат +7XXXXXXXXXX
- Данные пользователей: 152-ФЗ compliance
- Нет хранения паролей (только SMS OTP)

### Производительность
- proxy.ts: один SELECT по PK (session token) — максимально быстро
- Обновление session expires: throttle 1 раз в час
- Страницы кабинета: Server Components где возможно

## Non-Goals

- Социальная авторизация (Google, VK, Яндекс ID) — может быть позже
- 2FA (уже есть SMS — это и есть первый фактор)
- WebSocket / real-time обновления
- Push-уведомления
- Мобильное приложение
- Многоязычность (только русский)
- Оплата подписок (recurring) — только разовые платежи в MVP
- Email как метод входа
